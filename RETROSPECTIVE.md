## 2) Problem — 솔직한 한계

### Problem-1. AI 보조 기능 정확도 측정을 못 했음

- **사실**: `AiSummaryService.summarize` 는 Phase 2 단계 stub. 실제 Claude API 호출 X, 정확도 X.
- **왜 문제**: 면접에서 "AI 요약 정확도가 어땠나?" 라고 물으면 답할 측정값이 없음. "stub 이라 측정 못했다" 는 사실이지만 약함.
- **계획**: Phase 3 에서 사람 라벨링 50건 + AI 결과 일치율 + 사람 최종 판단 비율을 `evidence/ai-accuracy.md` 에 기록. 이력서에 "AI 보조 50건 검증, 일치율 X%" 한 줄 추가 가능하게.

### Problem-2. Sentry MCP 가 시연 단계에서 가치 제한

- **사실**: 실제 트래픽이 없어 Sentry 가 잡을 에러가 없음. MCP 연결 + DSN 환경변수까지만 시연.
- **왜 문제**: 라이프사이클 5 단계 중 "운영" 만 약함 → 발표·면접에서 다른 4 단계와 깊이 차이가 보임.
- **계획**: Phase 3 에서 의도적 NPE 1건 발생 → Sentry 에 잡힘 → Sentry MCP 에 위임 → root cause 후보 검증 → `MONITORING.md` 시나리오 섹션 보강.

### Problem-3. 5일 안에 8 PR 은 너무 타이트했음

- **사실**: Day 3 에 PR #3, #4, #5 가 동시 진행 → 의존성 chain 정리에 30분~1h 회의 추가.
- **왜 문제**: 4명 팀에서 PR 동시 진행 3개 이상 = 통합 비용 급증. 1개 충돌 발생 시 모두 멈춤.
- **계획**: 다음 기수 권장 = 5일에 PR 5~6개로 줄이기. 깊이 우선, 갯수 줄이기. 또는 6일 (토요일 1일 추가) 로 늘리기.

### Problem-4. 락 전략 결정 (D-006) 의 측정 근거가 없음

- **사실**: pessimistic vs optimistic 비교를 **로직만** 적었고 실측 부하 테스트 결과는 Phase 3 로 미룸.
- **왜 문제**: 면접에서 "왜 pessimistic 인가?" 답할 때 "사용자 100~1k 가정" 이라는 가정만 댈 수 있음. 측정 근거 없음.
- **계획**: Phase 3 hey + 동시 100 요청 시나리오로 두 락 전략 응답시간/실패율 비교 → `evidence/lock-comparison.md` 추가.
